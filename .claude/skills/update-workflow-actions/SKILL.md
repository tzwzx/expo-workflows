# Update Workflow Actions

`.github/` 配下のYAMLファイルで参照されている外部GitHub Actionsのバージョンを最新のメジャーバージョンに更新する。

## スコープ

- **対象**: `.github/` 配下すべての `.yml` / `.yaml` ファイル（`workflows/`、`actions/`、ルート直下問わず）
- **対象のアクション**: `uses:` で参照されている外部アクション（例: `actions/checkout@v6`、`oven-sh/setup-bun@v2`）
- **除外**: ローカルアクション（`./` で始まるパス参照）、Docker参照（`docker://`）、同一リポジトリの相対参照

## 手順

### 1. アクションの抽出

Globで `.github/**/*.{yml,yaml}` を列挙し、Grepで各ファイルから `uses:` 行を抽出する。抽出した文字列を `owner/repo@version` の形式にパースし、`owner`・`repo`・現行バージョンと、出現ファイルを一覧化する。

同じアクションが複数ファイルで使われている場合は、1件にまとめてファイル一覧を保持する。

### 2. 最新バージョンの確認

各アクションについて、**メジャーバージョンタグ**（`v6` のような形式）の最新を以下の優先順位で確認する：

1. **WebFetch** で `https://github.com/{owner}/{repo}/releases/latest` を取得
2. 取得できない場合は **WebSearch** で `{owner}/{repo} github action latest major version` を検索
3. それでも不明なら GitHub Marketplace ページや Releases 一覧を参照

複数のアクションがある場合は、並列に調査すると速い。情報量が多くなりそうな場合はサブエージェント（general-purpose）に `.github/` 配下すべてのアクションの最新バージョン調査をまとめて任せる。

**判断基準:**

- 現行がメジャーバージョンタグ（`v6` 等）なら、新しいメジャータグ（`v7` 等）が存在する場合のみ更新
- プレリリース・RC・ベータタグ（`v7.0.0-rc1` など）は除外。安定版の最新メジャーのみを採用
- 現行がコミットSHAやピン留め（`@abc123...`）の場合は原則触らず、レポートで「ピン留めされているため手動判断が必要」と注記する

### 3. ファイル更新

最新バージョンが現行と異なるアクションのみ、該当YAMLファイルの `uses:` 行を Edit で更新する。タグ表記スタイル（`v6` などのメジャーのみ）は現行と揃える。

### 4. レポート

以下の表形式で結果を出力する。更新が1件もない場合も表を出し、全件「（最新）」と記載する。

```
## 🔧 GitHub Actions 更新レポート

| アクション | 旧バージョン | 新バージョン | ファイル |
|-----------|------------|------------|---------|
| actions/checkout | v4 | v6 | .github/workflows/lint.yml, .github/workflows/test.yml |
| oven-sh/setup-bun | v2 | v2 | （最新） |
```

ファイル欄はプロジェクトルートからの相対パスで記載する。更新されたアクションは実ファイル名を列挙し、更新なしのものは「（最新）」と書く。

破壊的変更を含むメジャーバージョン更新を行った場合は、レポートの下に「⚠️ 破壊的変更の注意」セクションを設け、Releases から読み取った変更点の要旨を箇条書きで添える。ユーザーがワークフロー側の修正要否を判断できるようにするため。
